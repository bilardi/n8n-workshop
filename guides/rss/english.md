## Prerequisites

These steps must be completed before building the workflows.

- **For Workflow 7:** you need n8n running and Workflow 1 completed. If you haven't done it yet:
  1. Start from [intro/english.md](../intro/english.md) to create the Telegram bot and complete Workflow 1 (Part 4)
  2. Then follow the [Docker installation guide](../../scripts/docker_installer/english.md), or [server/english.md](../server/english.md) to access the shared server
- **For Workflow 8:** you also need Workflow 2 completed (Part 5 of the [Docker guide](../docker/english.md#part-5-workflow-2-polling-and-commands) or [server guide](../server/english.md)), because W8 reuses the same structure (Schedule + getUpdates + Switch)

### n8n API key

Workflow 8 queries the n8n REST APIs to check whether Workflow 7 is running. An API key is needed.

**Generate the key:**
1. In n8n: **Settings - n8n API - Create an API Key**
2. Copy the generated key

**Save it in the `.env` file** (in the `scripts/docker_installer/` folder):
```
N8N_API_KEY=the_copied_key
```

The `docker-compose.yml` already passes `N8N_API_KEY` to the container: no modifications needed.

**Restart the container:**
```bash
docker compose down && docker compose up -d
```

> On shared server: the `N8N_API_KEY` is unique for the entire n8n instance. It must be generated by the administrator (Settings - API) and set in the server's environment variables.

## Workflow 7 - RSS Fetch

Create a new workflow. For the Telegram node, copy the one already configured from a previous workflow and just change the text.

This workflow reads two RSS feeds in parallel, filters articles by keyword and saves the results. It introduces new nodes: **RSS Feed Read**, **Merge**, **Convert to/from File**, and **Write File to Disk**.

### Structure

```
[Schedule / Manual Trigger]
  ├── [RSS Feed Read: n8n blog]
  ├── [RSS Feed Read: Python blog]
  └── [Execute Command: prepare folder /tmp/workflow7]
         ↓
[Merge: combine all 3 outputs]
  ↓
[Code: filter by keyword]
  ↓
[Convert to/from File]
  ↓
[Write File to Disk: last_run.json]
  ↓
[Telegram: summary]
```

> [Workflow diagram](../../mermaid/english/workflow7.mermaid) | [Sequence diagram](../../mermaid/english/workflow.rss.mermaid)

### Node 1 - Schedule Trigger

- Interval: **1 hour** (or use Manual Trigger to test)

### Node 2a - RSS Feed Read (n8n blog)

- URL: `https://blog.n8n.io/rss/`

### Node 2b - RSS Feed Read (Python blog)

- URL: `https://blog.python.org/feeds/posts/default?alt=rss`

Connect both nodes directly to Node 1: n8n executes them **in parallel**. Each node returns one item per article with the fields: `title`, `link`, `pubDate`, `content`.

### Node 2c - Execute Command: prepare folder

```bash
mkdir -p /tmp/workflow7 && touch /tmp/workflow7/last_run.json && chmod 777 /tmp/workflow7/last_run.json
```

Also connect this node directly to Node 1, in parallel with the RSS nodes. Creates the folder and file with the correct permissions for the Write File to Disk node.

> **Why in parallel from the Trigger?** If the Execute Command were in sequence after the Code, the Convert to/from File node would inherit the Execute Command's binary data instead of the Code's JSON, producing a corrupted file. By placing it in parallel, the binary data doesn't interfere.

> **Alternative without Execute Command:** create the `/tmp/workflow7` folder directly in the Dockerfile (Docker) or ask the server administrator to create it with the correct permissions. In that case Node 2c is not needed.

### Node 3 - Merge

- Mode: **Append**
- Number of Inputs: 3

Connect the outputs of Node 2a, Node 2b and Node 2c to this node. The Merge combines all outputs: the RSS items from both feeds plus the Execute Command output. The following Code node filters only items that have a `title` field, ignoring the Execute Command output.

### Node 4 - Code: filter by keyword

- Mode: **Run Once for All Items**
- **Rename the node to "Code"** (double-click the node title in the editor): Node 7 (Telegram) references it as `$node["Code"]`

This node receives all RSS articles and returns a single JSON item with the filtered results. Define the keywords and feed at the top of the code.

Choose the language you prefer
- Javascript:
```javascript
const KEYWORDS = ["python", "n8n"];

const found = [];
for (const item of $input.all()) {
  const title = (item.json.title || "").toLowerCase();
  for (const kw of KEYWORDS) {
    if (title.includes(kw.toLowerCase())) {
      found.push({
        title:   item.json.title,
        url:     item.json.link,
        date:    item.json.pubDate,
        keyword: kw,
      });
      break;
    }
  }
}

return [{
  json: {
    timestamp: new Date().toISOString().slice(0, 19),
    keywords: KEYWORDS,
    items_found: found.length,
    items: found,
  }
}];
```
- Python:
```python
from datetime import datetime

KEYWORDS = ["python", "n8n"]

found = []
for item in _items:
    title = (item["json"].get("title") or "").lower()
    for kw in KEYWORDS:
        if kw.lower() in title:
            found.append({
                "title": item["json"].get("title"),
                "url":   item["json"].get("link"),
                "date":  item["json"].get("pubDate"),
                "keyword": kw,
            })
            break

return [{
    "json": {
        "timestamp": datetime.utcnow().strftime("%Y-%m-%dT%H:%M:%S"),
        "keywords": KEYWORDS,
        "items_found": len(found),
        "items": found,
    }
}]
```

> **Note:** The `datetime` import requires the native Python task runner (`n8n-task-runners` container with `N8N_RUNNERS_STDLIB_ALLOW=*` in the `n8n-task-runners.json` file). Without the runner, the Python Code node uses Pyodide which blocks all imports.

> **Tip for testing W8:** the W7 execution is very fast - W8 might not be able to see it as "running". To simulate a long execution, add a sleep in the Code node before the `return`:
> - Javascript: `await new Promise(r => setTimeout(r, 30000));`
> - Python: `import time` and `time.sleep(30)`

### Node 5 - Convert to/from File

- Operation: **Convert to File**
- Output format: **JSON**
- File name: `last_run.json`
- Mode: All items to One File

This node converts the JSON produced by the Code node into a binary file ready to be written to disk.

### Node 6 - Write File to Disk

- File Path: `/tmp/workflow7/last_run.json`

Overwrites the file on each execution: keeps only the latest result.

### Node 7 - Telegram, Send a text message

- Credential: your Telegram credential
- Chat ID: your fixed Chat ID (see Part 3 of the introduction)
- Text:
```
{{ $node["Code"].json.items_found > 0
  ? "📰 " + $node["Code"].json.items_found + " articles found (latest 10):\n" +
    $node["Code"].json.items.sort((a, b) => new Date(b.date) - new Date(a.date)).slice(0, 10).map(i => "• " + i.title + "\n  " + i.url).join("\n")
  : "📭 No articles found for: " + $node["Code"].json.keywords.join(", ") }}
```

> The template sorts articles by date (most recent first) and shows only the first 10. Telegram has a 4096-character limit per message: with 56 articles the message would be truncated.

> **Fixed Chat ID:** this workflow is started by a Schedule, not by a Telegram message, so there's no sender to extract the dynamic Chat ID from. Use the fixed Chat ID found in Part 3.

## Workflow 8 - Monitor

This workflow is the Telegram bot that responds to the `/status` command with the status of Workflow 7. The fastest way: copy the entire Workflow 6 (or Workflow 2), then modify only the `/status` branch.

```
[Telegram Trigger / Schedule+HTTP]
  ↓
[Switch: /status]
    ├── /status ──→ [HTTP Request: n8n API]
    │            └─→ [Read File from Disk] → [Convert to/from File: extract JSON]
    │                       ↓
    │               [Merge: combine API + file]
    │                       ↓
    │               [Code: format message]
    │                       ↓
    │               [Telegram]
    └── fallback → [IF regex] → [Telegram]
```

> [Workflow diagram](../../mermaid/english/workflow8.mermaid)

> **Why two parallel branches + Merge?** If the Read File from Disk receives input from the HTTP Request, it inherits its binary data and the Convert extracts those instead of the `last_run.json` content. By connecting them in parallel from the Switch, each node receives only its own data.

### Nodes 1-4 and Fallback

Configure **Node 1 (Schedule), Node 2 (HTTP Request getUpdates), Node 3 (Code filter recent), Node 4 (Switch), Node 5 (IF regex) and Node 6 (Telegram)** exactly as in [Workflow 2 - Part 5](../docker/english.md).

> On shared server use **Telegram Trigger** instead of Schedule + HTTP, as in [Workflow 6 - server](../server/english.md).

The only difference is in the `/status` branch of the Switch: use the following nodes.

### /status branch - Node A: HTTP Request (n8n API)

- **Rename the node to "HTTP Request"** (double-click the title): Node E (Code) references it as `$node["HTTP Request"]`
- Connect directly to the `/status` branch of the Switch node
- Method: **GET**
- URL: `http://localhost:5678/api/v1/executions`
- Send Query Parameters:
  - `workflowId`: the Workflow 7 ID (copy it from the editor URL after saving W7, e.g. `…/workflow/AbCdEfGh12` - `AbCdEfGh12`)
  - `status`: `running`
- Send Headers:
  - `X-N8N-API-KEY`: `{{ $env.N8N_API_KEY }}`

> On shared server: replace `localhost:5678` with the server domain.

> **The ID changes** if you delete and recreate Workflow 7. In that case update the `workflowId` parameter here.

Returns `{ data: [...] }` which will be an empty array if W7 is not running.

### /status branch - Node B: Read File from Disk

- File Path: `/tmp/workflow7/last_run.json`
- Connect directly to the `/status` branch of the Switch node (in parallel with Node A, not in sequence)
- **Set "Continue on Error"**: in the **Settings** tab of the node, **On Error** field, select **Continue**. If the file doesn't exist yet (W7 has never been executed), the workflow continues instead of stopping.

### /status branch - Node C: Convert from File

- Operation: **Extract from File**
- Input format: **JSON**
- **Set "Continue on Error"**: tab **Settings** > **On Error** > **Continue**. If Node B produced an error (file absent), this node also continues.

### /status branch - Node D: Merge

- Mode: **Append**
- Number of Inputs: 2

Connect the outputs of Node A (HTTP Request) and Node C (Convert) to this node. The Code will receive two items: item 0 = API data, item 1 = file data.

### /status branch - Node E: Code, format message

- Mode: **Run Once for All Items**

Choose the language you prefer
- Javascript:
```javascript
const items = $input.all();
const apiData = items[0].json.data ?? [];

let last = {};
const fileData = items[1].json.data ?? [];
if (Array.isArray(fileData) && fileData.length > 0) {
  last = fileData[0];
}

let msg;
if (apiData.length > 0) {
  msg = "🔄 Workflow 7 in progress";
} else if (last.timestamp) {
  const kw = (last.keywords || []).join(", ");
  msg = `✅ Workflow 7 idle\nLast execution: ${last.timestamp}\nArticles found: ${last.items_found ?? 0} (${kw})`;
} else {
  msg = "⏳ Workflow 7 not yet executed";
}

return [{ json: { messaggio: msg } }];
```
- Python:
```python
api_data = _items[0]["json"].get("data", [])

last = {}
file_data = _items[1]["json"].get("data", [])
if isinstance(file_data, list) and len(file_data) > 0:
    last = file_data[0]

if api_data:
    msg = "🔄 Workflow 7 in progress"
elif last.get("timestamp"):
    ts = last["timestamp"]
    n  = last.get("items_found", 0)
    kw = ", ".join(last.get("keywords", []))
    msg = f"✅ Workflow 7 idle\nLast execution: {ts}\nArticles found: {n} ({kw})"
else:
    msg = "⏳ Workflow 7 not yet executed"

return [{"json": {"messaggio": msg}}]
```

### /status branch - Node F: Telegram, Send a text message

- Credential: your Telegram credential
- Chat ID:
  - **Docker (Schedule+HTTP):** fixed Chat ID found in Part 3
  - **Server (Telegram Trigger):** `{{ $('Telegram Trigger').first().json.message.chat.id }}`
- Text: `{{ $json.messaggio }}`
